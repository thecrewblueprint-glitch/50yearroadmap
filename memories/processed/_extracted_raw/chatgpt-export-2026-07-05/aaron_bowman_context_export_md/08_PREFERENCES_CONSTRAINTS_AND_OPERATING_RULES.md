# Preferences, Constraints, and Operating Rules

## Output Style Preferences

Aaron strongly prefers:

- complete files
- complete code blocks
- full copy/paste-ready artifacts
- downloadable packages when possible
- no partial snippets
- no “insert this into your existing file” unless unavoidable
- direct, practical language
- step-by-step instructions when building something

Repeated instruction:

> Do not give half copies and tell me to inject them into another piece. Formulate the full copy-and-paste item.

## Project Workflow Preferences

Aaron benefits from:

- clear linear progression
- roadmap state tracking
- “you are now here” jumpers
- ability to move tasks later without losing them
- preventing idea sprawl
- keeping unfinished projects from being forgotten

## Resume / Professional Accuracy Rules

Do:

- present Deadhang Labor LLC as the applying entity when appropriate
- identify Aaron as owner/operator or primary contact
- describe work as live-event production labor and stagehand support
- mention relevant field exposure when appropriate

Do not:

- overclaim certifications
- imply direct employment/contracting/training unless confirmed
- list ARS as direct employer/client unless confirmed
- imply advanced rigging/climbing qualifications beyond what Aaron confirmed

## Technical Delivery Preferences

- User prefers ready-to-run/replacement files over fragments.
- User prefers zip/downloads for larger handoffs.
- User may use multiple AI tools and wants clean handoff prompts.
- User wants AI-agent prompts that are complete and enforce constraints.
- User wants GitHub work to avoid touching `main` unless specifically requested.

Aaron often asks for project/context memory to be preserved, but anything going into a public repo should be redacted first.
